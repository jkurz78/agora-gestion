# Revue de code — Compta V5 (fondations partie double)

- **Branche revue** : `feat/compta-v5` (39 commits, ~12 500 lignes)
- **Base** : `main` (`1b73690`)
- **Date** : 2026-05-30
- **Périmètre** : table `comptes`, écritures équilibrées (`EcritureGenerator`),
  lettrage (`LettrageService`), observer partie double, policies, migrations +
  seeders (système / bancaires / sous-catégories).

## Méthode

7 angles d'analyse croisés (scan ligne-à-ligne, comportement supprimé,
traçage cross-file, correctness comptable, isolation multi-tenant, qualité,
migrations/observer), puis vérification de chaque constat directement dans le
code source.

> ⚠️ **Risque transversal** : la suite Pest passe, **mais elle tourne sur
> SQLite** alors que la prod est MySQL. Les deux bloquants ci-dessous sont
> invisibles aux tests pour cette raison.

## 🔴 Bloquants

### #1 — `CAST(... AS INTEGER)` invalide en MySQL → migration `comptes` cassée en prod

`app/Services/Compta/Migrations/AuditGuard.php:80` (appelé sans branche driver
depuis `database/migrations/2026_05_20_000001_create_comptes_table.php:65`).

MySQL n'accepte pas `CAST(x AS INTEGER)` (il faut `AS SIGNED`) → erreur 1064.
Le docblock affirme à tort que « les deux moteurs acceptent CAST ». Le reste du
code (`BancairesSeeder`, `FluxTresorerieBuilder`) branche pourtant `isSqlite`
pour exactement ce cas. Tests verts sur SQLite, bug livré en prod : migration
000001 avortée, table `comptes` créée mais vide.

### #2 — `migrate:fresh --seed` laisse la table `comptes` vide

Les 3 seeders compta (`seedFromSousCategoriesSql`, `BancairesSeeder::seed`,
`SystemeSeeder::seed`) ne sont invoqués **que dans les migrations**
(000001/002/003), jamais rejoués par `DatabaseSeeder`. Or au moment des
migrations, `association` / `comptes_bancaires` / `sous_categories` sont vides
(peuplées ensuite par `DatabaseSeeder`). Résultat : `comptes` reste vide après
le bootstrap documenté → `Compte::ofNumeroSysteme('411'|'5112'|'530')` lève
`ModelNotFoundException`, tout `EcritureGenerator` est inopérant.

**Même problème pour tout nouveau tenant onboardé après les migrations** : pas
de hook qui re-seede les comptes système à la création d'une association.

### #3 — Lettrage sans invariant « même tiers » sur 411/401

`app/Services/Compta/LettrageService.php:50-58` — 5 invariants (même compte,
lettrable, équilibre, pas-de-relettrage, tenant) mais **aucun contrôle de
`tiers_id`**. Deux lignes 411 équilibrées de tiers différents peuvent être
lettrées ensemble → la créance du tiers A est soldée par le paiement du tiers
B, comptes auxiliaires corrompus. C'est précisément la convention « tiers sur
411/401 » posée par la spec (commit `b6aa0ea`).

### #4 — Remise bancaire : compte 512X résolu par IBAN (clé ni unique ni non-nulle)

`app/Services/Compta/EcritureGenerator.php:1093` —
`Compte::where('iban', $compteBancaire->iban)->first()`. Pas de colonne
`compte_bancaire_id` sur `comptes` : l'IBAN est l'unique clé de jointure. La
migration de backfill IBAN met `iban=''` sur les lignes existantes → avec
`iban=''`, plusieurs 512X matchent → `first()` débite sur le mauvais compte ;
avec `iban=NULL`, `WHERE iban = NULL` ne matche rien → exception alors que le
compte existe.

## 🟠 Importants

### #5 — `debit`/`credit` ajoutés `NOT NULL DEFAULT 0` sans backfill depuis `montant`

`database/migrations/2026_05_20_000004_*.php:41` — toutes les lignes historiques
passent à `debit=0 ET credit=0`, état que l'observer interdit. Seul le
discriminant `compte_id IS NULL` sauve. Couplage fragile : dès que le backfill
Step 32 pose `compte_id` sans renseigner `debit/credit` dans le même mouvement
Eloquent, l'observer lève et casse le backfill. Même défaut sur `equilibree` /
`type_ecriture` (000005, `DEFAULT 'normale'` → extournes legacy mal classées).

### #6 — Observer XOR : les montants négatifs passent

`app/Observers/TransactionLigneObserver.php:43-53` — `debit > 0 && credit > 0`
(faux si négatif) et `=== 0.0` (faux si négatif). Une ligne `debit=-50,
credit=0` est acceptée. L'invariant « débit XOR crédit, montant positif » n'est
pas garanti côté négatif ; deux négatifs compensatoires satisfont même
`assertEquilibre`. L'observer ne se déclenche pas non plus sur
`whereIn()->update()` / SQL brut, et aucune contrainte `CHECK` ne le double.

### #7 — Gardes tenant/tiers « no-op » si la relation `compte` n'est pas chargée

`EcritureGenerator.php:106, 145, 175` — `assertTenantCoherence`,
`assertTiersObligatoire411`, `assertPasDeTiersSur512` font `continue` quand
`relationLoaded('compte')` est faux. Les lignes **sources** récupérées via
`TransactionLigne::where(...)->first()` (encaissement créance, règlement
fournisseur, remise) n'ont pas la relation chargée → gardes neutralisées.

### #8 — `TransactionLigne` n'étend pas `TenantModel`

Modèle financier tenant-scopé sans scope global fail-closed (contraire à
CLAUDE.md). `LettrageService` fait `TransactionLigne::whereIn('id', $ids)
->update(...)` non scopé ; l'isolation repose entièrement sur les `assertTenant`
manuels et `whereHas('compte')`.

### #9 — `down()` migration 000004 : ordre `dropIndex` / `dropConstrainedForeignId` risqué (MySQL/InnoDB)

Les index composites `(compte_id, tiers_id[, lettrage_code])` peuvent servir de
backing-index à la FK `compte_id`. Les dropper avant la FK fait échouer
`migrate:rollback` en prod (errno 150).

## 🟡 Qualité / altitude (non bloquant)

- **Duplication massive dans `EcritureGenerator`** : le bloc
  `TransactionLigne::create([...]) + setRelation('compte', ...)` recopié ~14
  fois ; les 7 `pourXxx()` sont des jumeaux structurels. Un builder paramétré
  (sens + contrepartie + côté tiers + auto-lettrage) replierait le tout.
- **État redondant** : `debit` ET `credit` stockés (un toujours à 0) + `montant
  => 0` en dur alors que `montant_signe` le dérive ; `equilibree => true` écrit
  en dur *avant* `assertEquilibre`.
- **Règle « 512X bancaire »** (`classe===5 && str_starts_with('512') &&
  strlen>3`) réécrite à la main 4-5 fois alors que `Compte::scopeBancaires()`
  l'encode déjà (`512_%`).
- **`ComptePolicy`** ne vérifie pas `compte->association_id ===
  TenantContext::currentId()` (chemin `withoutGlobalScopes` / super-admin).
- **`lettrage_audit` insert** : `association_id`/`user_id` sans cast `(int)`.
- **`AuditComptaV5PreparationCommand`** : rapport JSON mono-fichier mélangeant
  tous les tenants, hors `storage/app/associations/{id}/`.
- **Pas de règlement partiel** sur `pourEncaissementCreance` (montant = débit
  source, solde toujours intégral).

---

## Verdict & priorisation

Socle solide et bien testé sur SQLite, mais **#1 et #2 empêchent le module de
fonctionner en prod / en environnement fraîchement seedé**, et **#3/#4** sont
des bugs comptables corrompant silencieusement les soldes.

Ordre de traitement recommandé :
1. Branche driver sur le `CAST` (#1)
2. Stratégie de seeding / onboarding des comptes (#2)
3. Invariant « même tiers » du lettrage (#3)
4. Clé stable banque ↔ compte (#4)
5. Backfill `debit/credit/equilibree/type_ecriture` (#5)
6. Durcissement des invariants observer / gardes (#6-#8)
